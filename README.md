## Краткое описание работы
В ходе выполнения лабораторной работы изучены принципы архитектуры REST и создано RESTful API с использованием контроллеров в ASP.NET Core. Освоена разница между Minimal API и Controller-based API, реализован полный набор CRUD-операций (Create, Read, Update, Delete) для управления задачами. Настроена документация через Swagger UI и протестированы все эндпоинты как через браузерный интерфейс, так и через расширение REST Client в VS Code. Также изучены правильные HTTP-статусы ответов, работа с DTO (Data Transfer Objects), маршрутизация с параметрами и фильтрацией, а также настройка CORS для кросс-доменных запросов.

## Итоговая таблица ASP.NET Core Controller-based API

| Аспект | ASP.NET Core Controllers |
|--------|-------------------------|
| Маршруты | [HttpGet] атрибут над методом |
| Группировка маршрутов | Класс-контроллер |
| Базовый URL | [Route("api/[controller]")] |
| Параметр пути | (int id) — параметр метода |
| Параметр запроса | [FromQuery] bool? completed |
| Тело запроса | [FromBody] CreateTaskDto dto |
| Ответ 200 | return Ok(data) |
| Ответ 201 | return CreatedAtAction(...) |
| Ответ 404 | return NotFound(...) |
| Ответ 204 | return NoContent() |
| Типизация данных | Строгая (C#) |
| Документация | Swagger — устанавливается отдельно |

## Главные выводы

1. **REST — это архитектурный стиль, а не протокол.** Те же принципы (клиент-сервер, stateless, единый интерфейс, кэшируемость) работают с любым языком программирования и фреймворком. Важно понимать, что REST определяет как организовывать взаимодействие, а не диктует конкретную реализацию.

2. **Контроллер в ASP.NET Core — это аналог Router в Express.js, но с мощными дополнительными возможностями.** В отличие от JavaScript, здесь есть автоматическая документация через Swagger, строгая типизация данных, встроенная валидация моделей и автоматическая сериализация в JSON. Это значительно ускоряет разработку и уменьшает количество ошибок.

3. **DTO (Data Transfer Objects) защищают API от некорректных данных.** Клиент может передать только те поля, которые сервер явно разрешает принять. Служебные поля (Id, CreatedAt, IsCompleted) устанавливаются сервером автоматически, что предотвращает манипуляции данными со стороны клиента.

4. **Swagger UI позволяет тестировать API без внешних инструментов.** Не нужно устанавливать Postman или писать тестовый JavaScript-код — вся документация генерируется автоматически на основе атрибутов контроллера, и каждый эндпоинт можно протестировать прямо в браузере с возможностью просмотра структуры запроса и ответа.

5. **Правильные HTTP-статусы — это часть контракта API.** Клиентское приложение (будь то браузер, мобильное приложение или другой сервис) должно понимать, что произошло, по коду ответа. 200 означает успех, 201 — создание ресурса, 204 — успешное удаление без тела ответа, 400 — ошибка валидации, 404 — ресурс не найден. Это позволяет клиенту корректно обрабатывать различные сценарии.

6. **CORS (Cross-Origin Resource Sharing) — критически важная настройка для разделения фронтенда и бэкенда.** Без правильной настройки CORS браузер заблокирует запросы с другого порта или домена. В разработке можно использовать AllowAnyOrigin(), но в продакшене необходимо указывать конкретные разрешённые источники.

## Примеры запросов

### Получить все задачи
```bash
curl http://localhost:5000/api/tasks
```

### Создать новую задачу
```bash
curl -X POST http://localhost:5000/api/tasks \
  -H "Content-Type: application/json" \
  -d "{\"title\":\"Новая задача\",\"description\":\"Описание\",\"priority\":\"High\"}"
```

### Обновить задачу
```bash
curl -X PUT http://localhost:5000/api/tasks/1 \
  -H "Content-Type: application/json" \
  -d "{\"title\":\"Обновлённая задача\",\"description\":\"Новое описание\",\"isCompleted\":true,\"priority\":\"Normal\"}"
```

### Удалить задачу
```bash
curl -X DELETE http://localhost:5000/api/tasks/1
```
